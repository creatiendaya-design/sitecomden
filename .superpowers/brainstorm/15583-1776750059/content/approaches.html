3 enfoques arquitecturales

Para el page builder + checkout COD. ¿Cuál encaja mejor con el proyecto?

A

Todo en el modelo Product (campos JSON)

Se agregan dos campos JSON directamente al modelo Product:

  • landingBlocks: Json — array de bloques con tipo, contenido y posición
  • checkoutMode enum — STANDARD | COD_ONLY | COD_AND_CART

Pros

  • 1 migración simple
  • Sin joins, todo en un fetch
  • Implementación más rápida

Contras

  • JSON no tipado en DB
  • Bloque individual no editable sin reescribir todo el array
B

Tabla separada LandingBlock ⭐ Recomendado

Nuevo modelo LandingBlock con relación a Product. Cada bloque es una fila con tipo, posición y JSON de contenido.

  • Product agrega checkoutMode + relación landingBlocks[]
  • CRUD individual por bloque (agregar, editar, eliminar, reordenar)

Pros

  • Bloques editables individualmente
  • Reordenamiento limpio por position
  • Escalable — se pueden añadir tipos de bloque nuevos

Contras

  • 1 query extra para los bloques
  • Migración + nuevo modelo
C

Modelos separados (ProductLanding + ProductCheckoutConfig)

Entidades completamente separadas de Product, vinculadas por FK.

  • ProductLanding — tiene los bloques y configuración visual
  • ProductCheckoutConfig — modo COD, settings del formulario

Pros

  • Separación máxima de responsabilidades
  • Posible reutilizar entre productos

Contras

  • Más modelos, más complejidad
  • Admin más fragmentado
  • Overkill para este caso

Mi recomendación: B — tabla separada da flexibilidad sin sobre-complicar. A es válido si se prioriza velocidad de implementación.
